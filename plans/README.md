# Airwave 2.0 Migration Plans

Generated by the `improve` skill on 2026-07-15 and extended through the
parametric-equalizer plans at commit `28b0210`.
Execute in order unless dependencies say otherwise. Each executor must read its
plan fully, honor STOP conditions, run every verification gate, and update its
status row.

## Non-negotiable product decisions

- Airwave 2.0 requires macOS 15 and processes the global stereo system mix.
- Airwave never changes the macOS default output or any device volume.
- The macOS default output and native volume are authoritative.
- Processing is always active while Airwave is running and ready; there is no
  routine engine toggle or remembered running state.
- Every stop, crash, permission loss, unsupported route, or runtime failure
  fails open to native unprocessed audio.
- Output changes are followed automatically with bounded retry and native audio
  during recovery.
- BlackHole, public/user-managed aggregate devices, and other virtual outputs
  are unsupported. A private aggregate containing Airwave's process tap is an
  internal Core Audio implementation detail only.
- The upgrade resets all Airwave preferences, onboarding, active preset, and
  Launch at Login registration, but never deletes user HRIR files.
- This is Airwave 2.0. Airwave 1.x remains the final macOS 14 line.

## Dependency notes

- 001 intentionally makes the branch temporarily unable to process audio. The
  user chose removal of legacy routing before replacement work so the dangerous
  route and volume mutations disappear first.
- 002 defines fakeable boundaries and safety tests before 003 touches HAL APIs.
- 003 owns low-level resource lifecycle; 004 owns policy, retries, and output
  changes. Do not merge those responsibilities into a new god object.
- 005 consumes the stable runtime state from 004; it must not invent a second
  readiness or permission state machine.
- 006 is the release gate. Performance measurements are diagnostic only; safety
  and functional checks block release.
- 007 is research-only. It may authorize a future implementation plan, but it
  must not add multichannel behavior to the 2.0 stereo runtime.
- 008 is deliberately presentation-only. It establishes the shared animated
  preset component and two-column Settings shell without changing persistence,
  filesystem behavior, or application lifecycle.
- 009 depends on 008 so drag targeting and preference rows reuse the completed
  visual components. It owns all filesystem mutation, menu visibility,
  LSUIElement reopen behavior, and their higher-risk validation.
- 010 establishes a strict, atomic managed-file boundary before untrusted text
  can become realtime DSP configuration. It reuses 009's import conventions.
- 011 depends on 010's format-neutral model and is deliberately DSP-only so
  coefficient fidelity and render-thread safety can be reviewed in isolation.
- 012 composes the tested processor with HRIR and changes readiness to “any
  effect” without widening the Core Audio platform or volume/route authority.
- 013 depends on the completed library and runtime so SwiftUI remains a thin
  import/select/inspect surface rather than a second state machine.
- 014 is a focused correctness repair on completed plans 004 and 005. It keeps
  persisted onboarding separate from transient runtime health and must pass a
  signed device-connect/relaunch matrix before retaining the uncommitted Core
  Audio I/O-proc rewrite.

## Findings considered and rejected

- Preserve a legacy routing fallback: rejected because any shipped path that
  can mutate default output or physical volume violates the safety guarantee.
- Require macOS 26 for newer tap conveniences: rejected because macOS 15 has the
  required process-tap API and macOS 26 does not remove the core HAL lifecycle.
- Preserve legacy settings or silently migrate: rejected; all preferences and
  onboarding reset, while user HRIR files remain untouched.
- Make latency or CPU thresholds release blockers: rejected by product choice;
  record measurements, but only safety and functional failures block 2.0.
- Implement per-application capture in 2.0: rejected; 2.0 captures all system
  audio except Airwave itself.
- Bundle or download EQ curves: rejected for v1; Airwave provides only the
  synthetic None selection and users import their own EqualizerAPO text files.
- Add EQ editing, response graphs, a limiter, or automatic headroom: rejected
  for v1; imported values and preamp are applied exactly and remain read-only.
- Support full EqualizerAPO configuration files: rejected for v1; only Preamp
  and Q-based PK/LSC/HSC filters are accepted atomically.
